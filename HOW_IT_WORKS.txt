1) APP OVERVIEW
================================================================================

This app is a Streamlit trading platform with 5 pages:

1. Live Analysis
2. Performance
3. Prop Firm
4. MT5 Auto-Trade
5. Grid Trading

Core design principle:
- Heavy analytics and backtesting belong to analysis pages.
- MT5 Auto-Trade must stay lightweight and prioritize live execution timing.
- Production mode uses a separate background worker process for execution.

================================================================================
2) PAGE FLOW
================================================================================

A) Live Analysis
- Fetches OHLCV by symbol and timeframe.
- Generates strategy signals from selected ICT concepts.
- Renders TradingView-style chart and signal table.
- Intended for research and setup validation.

B) Performance
- Runs broad backtests across mode timeframes.
- Computes statistics, equity curve, drawdown, R:R sweeps.
- Intended for optimization and parameter selection.

C) Prop Firm
- Applies risk rules (drawdown and daily loss constraints).
- Simulates account behavior and position sizing.
- Intended for challenge/risk management validation.

D) MT5 Auto-Trade (LIVE CRITICAL)
- Connects to MT5 and streams account/position status.
- Monitors only fresh live-window signals.
- Executes fresh OPEN signals at live market price.
- Ignores stale historical setups.
- Enforces execution guardrails (max spread, max entry drift, kill-switch).
- Enforces execution policy (max trades/day, max open positions, UTC session window).
- Enforces signal quality gate (min quality score + max signal age in bars).
- Records all outcomes in a persistent audit log.
- Provides Worker Control to run execution in execution_worker.py.
- Includes a dry-run LONG execution test to validate broker path without placing an order.

E) Grid Trading (NEW)
- AI analyzes market direction using ICT signals + EMA(20/50) + ATR regime.
- Builds a grid of small-lot BUY and SELL levels around current price.
- AI direction bias (BULLISH / BEARISH / NEUTRAL) skews grid level allocation.
- Dynamic regime behavior:
  * Sideways (RANGING): tighter spacing and denser symmetric grid.
  * Trending (TRENDING): wider spacing and stronger directional skew.
  * Volatile (VOLATILE): widest spacing, fewer levels, reduced lot pressure.
- Lot scaling: slightly larger toward bias, smaller on counter-direction levels.
- ATR-based dynamic spacing; Brain auto-tunes spacing multiplier over time.
- GridBrain learns from every closed trade (grid_audit_log.jsonl).
- Brain tracks: spacing, TP/SL, max open levels, directional edge by regime.
- Self-learning activates after 5+ trades per symbol/regime bucket.
- Grid state is persisted to grid_state.json across restarts.
- Manual level close and kill-switch (Deactivate Grid) for risk control.
- Hard risk guards: max daily loss, max peak drawdown %, optional minimum equity.
- Auto profile switch by regime: RANGING / TRENDING / VOLATILE parameter sets.
- Basket manager closes all OPEN grid levels at basket TP/SL thresholds.
- Trailing Basket TP: once basket PnL exceeds TP, keeps position open while
  PnL rises.  Places a trailing stop at (peak_pnl - trailing_step_usd).
  Closes only when PnL drops below the trail floor — locking in more profit.
- Session-aware grid pause: optionally pauses new fills during low-liquidity
  sessions (e.g., Asian) while continuing to monitor existing positions.
- Grid P&L Equity Curve: cumulative equity chart from closed levels in audit log.
- Built-in grid backtest includes spread/slippage and monthly walk-forward summary.
- Execution mode safety: Dry-run default, live order mode requires explicit user confirmation.
- Backtest results can be exported as CSV (summary, equity curve, monthly walk-forward).
- Account safety presets (2k/5k/10k) auto-apply conservative lot and risk limits.
- Portfolio-wide caps can block fills by total open positions and total lots.
- Daily audit report export supports UTC date filtering and CSV download.
- Dynamic Plan Preview panel shows computed buy/sell counts, effective spacing,
  lot pressure, and regime parameters BEFORE grid activation.
- Hard lockout: risk guard breach persists a lockout to disk (lockout_active,
  lockout_reason, lockout_time) — survives restarts, requires manual unlock
  with a reason that is logged to the audit trail.

Performance optimizations:
- Brain retraining moved from every page load to on-demand button only.
- Duplicate regime analysis removed — single EMA/ATR computation per refresh.
- MT5 Ops summaries throttled: all aggregations skipped on fast-cycle refreshes.
- Chart ICT detectors guarded by show_* toggles (only run when enabled).
- Grid data fetching cached with TTL=30s to avoid redundant API calls.

Cloud runtime note:
- Streamlit Community Cloud is Linux, so direct MT5 terminal execution is not supported there.
- Cloud app is for dashboard/monitoring; live MT5 execution must run on Windows (local/VPS).

3) DATA AND CACHING MODEL
================================================================================

Historical data cache:
- load_data(): cached by Streamlit with TTL tied to refresh interval.
- Used by Live Analysis, Performance, and Prop Firm.

Historical signal cache:
- load_signals(): full signal generation on cached OHLCV.
- Used by analysis/backtest pages and on-demand AI sizing.

Live signal cache (new optimization):
- load_live_signals(): short TTL (10s), recent bars only.
- Uses limited rolling window instead of full history.
- Filters to recent signal times only.
- Used by MT5 Auto-Trade so auto-refresh remains fast.

Result:
- Auto-refresh no longer spends time recalculating full historical signal sets
  during live execution checks.
- Execution can run independently from Streamlit reruns via worker files.

Worker runtime files:
- worker_config.json: worker strategy + polling settings.
- worker_state.json: executed signal keys (duplicate prevention).
- worker_heartbeat.json: last worker status and timestamp.

4) MT5 AUTO-TRADE EXECUTION LOGIC (UPDATED)
================================================================================

Step 1: Connect and validate MT5 account
- Loads credentials (optional saved config).
- Connects and displays balance/equity/PnL/positions.

Step 2: Pull live market price
- Reads bid/ask and tick time for the configured MT5 symbol.

Step 3: Build live signal window only
- Generate signals from recent bars (not full history).
- Keep only very recent signal timestamps.
- Focus on status OPEN and freshness window.

Step 4: Identify tradable signal
- Signals are ranked by quality score.
- Quality gate enforces minimum score and maximum age (bars).
- Best eligible OPEN signal is selected for execution.

Step 5: Execute at market
- Manual or auto mode places market order via execute_trade().
- Dry-run test uses preview_trade_execution() to validate order path without placing a live order.
- SL/TP are recalculated from live price using signal distance offsets.
- Broker minimum stop distance is enforced before order send.
- Trade is blocked if spread/entry drift exceeds configured thresholds.
- Kill-switch can block all new entries instantly.
- Pre-trade SL distance validation (new guardrails):
  * Rejected if SL distance is inside the spread (negative or zero net distance).
  * Rejected if SL is tighter than min_stop_distance_pips (default: 2 pips).
  * Rejected if SL is wider than max_stop_distance_pips (default: 200 pips).
  * Rejected if TP/SL ratio is below min_rr_ratio (default: 1.5).
  * All thresholds are configurable from the Execution Guardrails panel.

Step 6: Duplicate prevention
- Each signal execution uses a unique session key.
- Prevents repeated re-execution on every refresh cycle.

Step 7: Persistent audit trail
- FILLED, FAILED, and BLOCKED events are appended to mt5_audit_log.jsonl.
- MT5 page shows both current session log and persistent recent history.

Step 8: Policy engine checks
- Max trades/day: blocks new entries after daily FILLED limit.
- Max open positions: blocks entries when symbol position cap is reached.
- Session window (optional): blocks entries outside allowed UTC hours.

Step 9: Background worker mode
- Save worker config from MT5 page (symbol/timeframe/concepts/rr/lookback/lot).
- Start worker process from terminal:
  .venv\Scripts\python.exe execution_worker.py
- MT5 page now has a persistent Execution Profile (style/symbol/timeframe/concepts/RR/lookback).
- Execution Profile values stay fixed across auto-refresh and page reruns until manually changed.
- Worker polls live window signals and executes only fresh OPEN signals.
- Worker uses quality gate defaults: min_quality_score=68 and max_signal_age_bars=2.0.
- In-app auto-trade is locked when worker mode is enabled and heartbeat is healthy (anti-duplicate).
- If worker mode is enabled but heartbeat is stale/missing/error, in-app auto-trade can be used as fallback.

Step 10: Worker operations and monitoring
- Quick start script: start_worker.ps1
- Quick stop script: stop_worker.ps1
- Worker heartbeat staleness alert appears in MT5 page.
- MT5 page supports dedicated real-time refresh cadence (1s/2s/3s/5s/10s/30s) independent of global app refresh.
- Fast cadence improves PnL/order visibility and quicker fresh-signal reaction, with higher CPU/network cost.
- Optional Lean MT5 UI mode auto-optimizes ultra-fast cadence by hiding heavy tables/charts.
- Lean mode prioritizes live prices, open-trade metrics, scanner status, and execution actions.
- Live Signal Monitor now shows a Digital Scanner status loader each cycle.
- Scanner displays phase/cycle progress while refreshing live-window signal checks.
- Scanner includes a digital stream tape and live status chips (phase, symbol, timeframe).
- Market Condition Analysis panel summarizes regime, directional bias, conviction, structure state, execution risk, and spread.
- Market Condition Analysis also classifies market phase (Trend/Pullback/Chop) from recent signal sequence.
- Spread risk badge now shows spread-limit utilization percent with color-coded severity.
- Regime score sparkline tracks the last 10 scan cycles for quick momentum context.
- Execution Ops section shows 24h Filled/Failed/Blocked and fill rate.
- Execution Ops includes broker response KPIs (Avg ms and P95 ms).
- Execution Ops shows hourly status trend chart for FILLED/FAILED/BLOCKED behavior.
- Last Execution Attempt panel shows latest status/symbol/direction/time and reason text.
- Rejection Reason Breakdown table summarizes top FAILED/BLOCKED causes in last 24h.

Step 11: Watchdog auto-restart
- Quick start script: start_watchdog.ps1
- Quick stop script: stop_watchdog.ps1
- Watchdog monitors worker_heartbeat.json and restarts worker on stale/error state.
- Watchdog status appears in MT5 page from watchdog_heartbeat.json.

Step 12: Policy profiles
- Conservative: tighter spread/drift and lower trade caps.
- Balanced: default production profile.
- Aggressive: wider spread/drift and higher trade caps.
- Apply Profile button updates execution guardrails and policy instantly.

Step 13: Webhook alerting
- Configure alerting from MT5 page (enable, webhook URL, cooldown, alert types).
- Worker sends alerts for connect errors, exceptions, and trade failures.
- Watchdog sends alerts when it triggers a worker restart.
- Test alert button validates webhook integration.

Step 14: Auto-start on reboot (Windows)
- Install scheduled tasks: install_scheduled_tasks.ps1
- Remove scheduled tasks: uninstall_scheduled_tasks.ps1
- Tasks created:
  - SacredSoulBot-ExecutionWorker
  - SacredSoulBot-WorkerWatchdog

Step 15: Live validation status (implemented)
- Dry-run execution path validated (preview mode).
- Real sim-account manual execution validated (open + immediate close).
- Worker runtime validated (heartbeat + monitoring + audit updates).

5) WHY EXECUTION IS NOW MORE RELIABLE
================================================================================

Previous risk:
- Every refresh triggered heavy backtest-style calculations.
- Execution checks could be delayed under load.

Current behavior:
- MT5 page uses a live-only monitor path.
- Historical analytics are separated from execution path.
- AI lot sizing is on-demand, not auto-run each refresh.
- Hard guardrails block bad conditions before order_send.
- Persistent audit improves traceability and support/debug workflows.

Outcome:
- Faster refresh cycles for live trading.
- Lower chance of missing timely MT5 executions.

8) GRID V2 RISK + LEARNING FLOW
================================================================================

Step G1: Regime detection and profile selection
- AI classifies current market as RANGING, TRENDING, or VOLATILE.
- If Auto Regime Profile is enabled, grid settings are switched automatically:
  * Range profile: tighter spacing, more open levels.
  * Trend profile: wider spacing, fewer open levels.
  * Volatility profile: widest spacing, strict open-level cap.

Step G2: Brain-assisted parameter tuning
- Brain recommendations are loaded by symbol + regime bucket.
- If sufficient trade samples exist, recommended values are applied:
  * spacing multiplier
  * TP multiplier
  * SL multiplier
  * max open levels

Step G3: Hard risk guard checks (every refresh cycle)
- Pulls balance/equity from MT5 account.
- Blocks and auto-deactivates grid when any condition is breached:
  * daily loss >= max_daily_loss_usd
  * drawdown from peak equity >= max_drawdown_pct
  * equity <= min_equity_usd (if configured)

Step G4: Basket-level control
- Computes floating basket PnL across all OPEN levels.
- If basket PnL reaches configured threshold:
  * basket_take_profit_usd => close all open levels
  * basket_stop_loss_usd => close all open levels

Step G5: Learning data capture
- Closed level events append rich context fields to grid_audit_log.jsonl:
  * symbol, regime, direction, lot, spacing_used
  * TP/SL multipliers, max_open_levels, timeframe, profile_name
- GridBrain retrains from audit log and updates memory file.

Step G6: Backtest and walk-forward validation
- Grid backtest uses OHLC simulation with spread + slippage haircut.
- Includes periodic rebalance to refresh regime/profile.
- Outputs:
  * summary metrics (trades, win rate, net PnL, max DD)
  * equity curve
  * monthly walk-forward table

Grid runtime files:
- grid_state.json: active grid levels + risk tracking state.
- grid_audit_log.jsonl: closed-trade events for analytics/learning.
- grid_brain_memory.json: learned recommendations per symbol/regime.
